# Markean Beta Backend and Auth Rollout Design

Date: 2026-04-11

## Summary

Markean will ship its first small-beta backend on Cloudflare. The product will use a Cloudflare Worker as the public API, D1 as the system of record, Durable Objects for sync coordination, and R2 for exports and backups. Authentication will support Google login, Sign in with Apple, and email magic links, but the application session and user records will remain owned by Markean rather than a third-party auth platform.

This rollout is intentionally optimized for a small beta rather than a public launch. The design favors a simple, teachable operational model that the operator can execute manually and understand step by step. Domain, DNS, OAuth console, and Apple developer changes should be performed by the operator with guided instructions rather than being auto-applied by the implementation.

## Product Goals

- Put Markean on the cloud for real beta use.
- Support both web and Expo mobile clients.
- Keep business data, users, and sessions in Markean-owned infrastructure.
- Use Google and Apple only as identity providers, not as the long-lived application session.
- Add email magic links as a fallback login path.
- Preserve the existing local-first sync model.
- Teach the operator how each infrastructure step works instead of hiding the setup behind one-click automation.

## Constraints and Assumptions

- The public web app will live at `https://markean.mizutani.top`.
- The public API will live at `https://api-markean.mizutani.top`.
- The existing blog and other services on `www.mizutani.top` must remain untouched.
- The operator will manage Cloudflare DNS and domain attachment manually.
- The current repo already contains `apps/api` as a Cloudflare Worker backed by Hono, D1, and Durable Objects.
- The current repo does not yet contain a committed Expo app in this worktree, but the backend must reserve a correct auth flow for an Expo client.
- Small beta access will stay limited through an approved-email allowlist. Successful provider login proves identity, but the backend will only create an application session for approved beta users.

## Current Starting Point

The current repository already has the right backend foundation:

- `apps/api`
  - Hono Worker entry point
  - D1 binding
  - Durable Object sync coordinator
  - bootstrap and sync routes
- `apps/web`
  - Vite web client
  - local-first browser storage
  - background sync scaffolding
- shared packages
  - domain models
  - API client
  - sync core
  - browser storage

The main missing pieces for beta readiness are:

- real authentication flows
- production environment separation
- stable session handling for web and mobile
- cloud resource naming and deployment rules
- operational documentation for manual setup

## Chosen Approach

Markean will use a self-managed auth/session model on top of Cloudflare:

- Google login via OAuth 2.0 / OpenID Connect
- Apple login via Apple Sign In
- email magic links generated by the Markean API
- email delivery handled by Resend for the beta rollout
- application sessions created and validated by Markean in D1

This is the right balance for beta because:

- it keeps user and session ownership inside Markean
- it avoids the complexity of building email delivery from scratch
- it supports both browser and Expo clients cleanly
- it is operationally small enough for one person to learn and run

### Environment Model

Two environments are required from the start:

- `dev`
  - local and development testing
  - temporary credentials
  - dev-only helpers such as `/api/dev/session`
- `prod`
  - real beta traffic
  - real OAuth credentials
  - real mail provider keys
  - production D1 and R2 resources

The implementation should treat `dev` and `prod` as different configurations, not as one deployment with a few renamed variables.

## Auth and Session Design

### Identity Providers

Supported login methods:

- Google
- Apple
- email magic link

Provider login proves who the user is. It does not become the long-lived Markean session.

### Web Session Model

The web app will use an opaque session token in a secure `HttpOnly` cookie set by the API domain.

Rules:

- cookie is created only by the API
- cookie is never readable by frontend JavaScript
- cookie is `Secure`
- cookie is `SameSite=Lax`
- frontend requests use `credentials: include`
- API CORS must explicitly allow `https://markean.mizutani.top`

The cookie should be host-scoped to `api-markean.mizutani.top` rather than shared with every subdomain under `mizutani.top`.

### Expo Session Model

The Expo app will not depend on browser cookies. It will use a short auth-code exchange followed by a Markean-issued mobile session token.

Flow:

1. Expo opens the provider login in the system browser.
2. Provider returns to the Markean API callback.
3. The API validates the identity and creates a short-lived one-time auth code.
4. The API redirects to the Expo deep link with the auth code.
5. Expo calls `POST /api/auth/exchange`.
6. The API returns a Markean mobile session token.
7. Expo stores that token in `SecureStore`.
8. Future mobile API calls send `Authorization: Bearer <token>`.

This keeps long-lived mobile sessions out of URL fragments and gives the backend one consistent place to create and revoke mobile sessions.

### Email Magic Links

Email magic links are not OAuth. They are a Markean-managed login flow.

Flow:

1. User submits email.
2. API validates whether the email is approved for beta access.
3. API creates a single-use token with a short expiration.
4. API sends the magic link email through the mail provider.
5. User opens the link.
6. API verifies the token and either:
   - sets the web session cookie and redirects to the web app
   - or issues a one-time auth code for Expo exchange

Magic link tokens must be:

- random
- single-use
- short-lived
- stored as hashes in D1 rather than plain text

### Beta Access Rule

For a small private beta, authentication does not imply automatic product access. The backend will enforce an allowlist based on approved email addresses. This keeps Google, Apple, and magic link sign-in compatible with a private beta rollout without requiring a separate invite-only identity provider.

## Data Model Changes

The existing D1 schema should be extended with the smallest set of auth tables required for beta.

### `auth_accounts`

Purpose:

- map a Markean user to Google or Apple identities

Suggested fields:

- `id`
- `user_id`
- `provider`
- `provider_subject`
- `email`
- `email_verified`
- `created_at`
- `updated_at`

### `sessions`

The current session table should evolve so it can represent both web and mobile sessions.

Suggested additions:

- `token_hash`
- `client_type` (`web` or `mobile`)
- `created_at`
- `expires_at`
- `revoked_at`

### `magic_link_tokens`

Purpose:

- track email magic link issuance and consumption

Suggested fields:

- `id`
- `email`
- `token_hash`
- `client_type`
- `redirect_target`
- `expires_at`
- `consumed_at`
- `created_at`

### `beta_allowed_emails`

Purpose:

- `beta_allowed_emails`

This table can stay deliberately simple and is required for the beta rollout.

## API Surface

The Worker should expose a focused auth surface:

- `GET /api/auth/google/start`
- `GET /api/auth/google/callback`
- `GET /api/auth/apple/start`
- `GET /api/auth/apple/callback`
- `POST /api/auth/email/request`
- `GET /api/auth/email/verify`
- `POST /api/auth/exchange`
- `POST /api/auth/logout`
- `GET /api/me`

Existing app APIs such as `/api/bootstrap` and `/api/sync/*` should move behind shared auth middleware that accepts either:

- a valid web session cookie
- or a valid bearer token for mobile

## Deployment and Routing Design

### Web

- deployed to `markean.mizutani.top`
- talks to `api-markean.mizutani.top`
- does not share its own origin with the API

### API

- deployed as a Worker on `api-markean.mizutani.top`
- owns auth callbacks
- owns cookies
- owns API CORS policy

### Why Subdomains Instead of a Path Prefix

Subdomains are chosen over `www.mizutani.top/markean` because they reduce operational friction:

- OAuth callback URIs are cleaner
- API routing is separated from the blog
- cookies are easier to scope safely
- Expo callback and backend URLs stay stable
- future deployment and debugging are simpler

## Manual Rollout Order

The implementation and operator guide should follow this order exactly.

### Phase 0: Preserve Safe Working Conditions

- continue backend work in a dedicated git worktree
- do not modify the branch actively used for UI and UX changes

### Phase 1: Cloudflare Production Resources

- confirm the Cloudflare zone for `mizutani.top` is available
- create the production Worker
- create the production D1 database
- create the production R2 bucket
- configure Durable Object migrations

### Phase 2: Production Domain Attachment

- attach `api-markean.mizutani.top` to the Worker
- attach `markean.mizutani.top` to the web deployment target
- verify DNS records manually
- verify HTTPS is active on both subdomains

### Phase 3: Environment Variables and Secrets

Required production settings should include:

- `APP_ENV`
- `APP_BASE_URL`
- `API_BASE_URL`
- `GOOGLE_CLIENT_ID`
- `GOOGLE_CLIENT_SECRET`
- `APPLE_CLIENT_ID`
- `APPLE_TEAM_ID`
- `APPLE_KEY_ID`
- `APPLE_PRIVATE_KEY`
- `MAGIC_LINK_SECRET`
- `MAGIC_LINK_TTL_MINUTES`
- `EMAIL_FROM`
- `EMAIL_PROVIDER_API_KEY`

`ALLOW_DEV_SESSION` must remain disabled in production.

### Phase 4: Google OAuth Setup

- create a Google OAuth app
- register the production callback URI
- register the development callback URI
- store the credentials in Cloudflare secrets
- verify login from web first

### Phase 5: Apple Sign In Setup

- create or configure the Apple Sign In service ID
- register the production callback URI
- register the development callback URI
- create the Apple signing key
- store the Apple credentials in Cloudflare secrets
- verify login from web first

Apple setup is operationally harder than Google. The operator guide must call that out clearly and keep the steps explicit.

### Phase 6: Mail Delivery Setup

- create and configure the Resend account
- verify the sending domain or sender identity
- configure the production API key
- send a real test magic link
- verify expiration and single-use behavior

### Phase 7: Expo Integration

- define the Expo deep link scheme
- register mobile redirect URIs
- implement auth code exchange
- verify Google, Apple, and magic link flows on device
- store mobile session tokens only in `SecureStore`

### Phase 8: Beta Verification

- test login with at least two different beta users
- verify data isolation across users
- verify web logout
- verify mobile logout
- verify expired session behavior
- verify sync still works after authentication is enforced

## Error Handling and Observability

Auth and sync failures should be easy to identify in logs.

### Error Response Shape

API error responses should include:

- `code`
- `message`
- `requestId`

### Required Logs

- auth start
- auth callback success
- auth callback failure
- magic link request success
- magic link request failure
- session creation
- session revocation
- sync push success
- sync conflict
- sync failure

### Required Log Dimensions

- `requestId`
- `userId` when known
- `provider`
- `clientType`
- timestamp

## Acceptance Criteria

The beta backend rollout is complete only when all of the following are true:

1. Web login works with Google.
2. Web login works with Apple.
3. Web login works with email magic links.
4. Expo login works with Google through auth-code exchange.
5. Expo login works with Apple through auth-code exchange.
6. Expo login works with email magic links.
7. `GET /api/bootstrap` returns only the current signed-in user's data.
8. Existing sync endpoints reject missing or invalid sessions with `401`.
9. Different users cannot read each other's notes.
10. Production does not expose `/api/dev/session`.
11. The operator has a written manual checklist for Cloudflare, Google, Apple, and mail setup.
12. The unit test suite passes.
13. The configured end-to-end test suite passes.

## Out of Scope

- full public self-serve signup
- collaboration or shared workspaces
- attachments
- advanced RBAC
- analytics or billing
- replacing D1 and Durable Objects with a different backend stack
- handing long-lived session authority to an external auth platform

## Implementation Guidance

The follow-up implementation plan should prioritize:

1. production-safe config and environment separation
2. D1 auth schema changes
3. shared auth middleware
4. web login flow
5. Expo auth-code exchange flow
6. magic link email flow
7. production rollout checklist and operator guide

The implementation plan should also preserve the existing sync architecture rather than refactoring it during auth work.
